# 代码生成 Wiki 文档（通用）

## 概述

**目的**：将代码库（或其子集）分析并转化为结构化的 Wiki 文档，便于团队理解与检索。

**功能**：
- 调研代码库结构与核心模块（受排除项约束，避免全盘扫描）
- 先生成大纲并落盘，经用户确认后再细化撰写
- 多页面时通过 task-dispatch 并行加速撰写
- 按统一格式规范撰写 Wiki 页面（含源码引用），并自检 R1-R6
- 每生成一篇 Wiki 后**必须调用格式检查命令**校验格式（见"格式检查脚本"一节），不合规需修复至通过

**使用场景**：
- 用户要求"根据代码生成 wiki"、"为这个项目写文档"、"梳理 XX 模块"
- 用户想把某个目录/模块整理成可检索的知识文档
- 新成员 onboarding，需要代码库地图

> 本技能**不限定项目**。Wiki 格式规范内联于下文，作为单一权威定义。若目标环境已安装 CodeToWiki（提供 `codetowiki wiki-format` 命令），**生成 Wiki 后必须调用该命令做格式校验**（见"格式检查脚本"一节）；未安装时，退化为按 R1-R6 手动自检，不得跳过校验环节。

## 前置确认（必须）

在动手前，**必须**与用户确认以下信息（不要假设、不要自己决定存储位置）：

1. **Wiki 存储根目录**：用户指定的绝对路径（如 `/path/to/wiki`）。目录不存在则创建。
2. **分析范围**：整个仓库 / 指定目录 / 指定模块 / 指定文件列表。
3. **排除路径**：哪些代码/目录不纳入分析（见"排除路径"一节，用户可增删）。
4. **是否已有 Wiki**：目标目录是否已有内容需保持风格一致，还是全新创建。
5. **拆分粒度**：单篇覆盖 vs 总览+子篇拆分（默认：文件数 ≥ 5 建议拆分）。
6. **语言**：Wiki 正文语言（默认与用户沟通语言一致）。

> ⚠️ 禁止在未经确认的情况下自行决定 Wiki 存储位置。存储位置是必填项，未确认前不得写入任何文件。

## 排除路径（避免全盘扫描）

调研阶段**严格跳过**以下类型，只扫描业务源码。用户可按项目增删：

| 类别 | 默认忽略 |
|------|----------|
| 依赖/构建产物 | `node_modules/`、`vendor/`、`dist/`、`build/`、`target/`、`__pycache__/`、`*.pyc`、`*.min.js`、`*.bundle.js` |
| 测试与迁移 | `*/tests/`、`*/test/`、`*/migrations/`、`*/__init__.py` |
| 第三方/子模块 | `third_party/`、`submodules/`、各语言包缓存（如 `.venv/`、`site-packages/`） |
| 已有文档 | `docs/`、`**/*.md`（避免重复文档化已有文档） |
| VCS/CI | `.git/`、`.github/`、`.gitlab-ci.yml`、`.ci/` |
| 生成代码 | `*.pb.go`、`*_gen.py`、`*.gen.go`、`swagger.json`、`openapi.json` 等自动生成物 |

> 排除项同时用于"代码调研"和"大纲生成"，确保不把噪音文件当作分析对象，也不为其建 Wiki 页。

## Wiki 格式规范（单一权威定义）

所有生成的 Wiki 页面必须遵守以下规范。本节是格式规范的权威定义；强制校验由下文"格式检查脚本"一节执行。

### 页面结构模板

```markdown
# 标题

<cite>
**本文引用的文件**
- [模块名](file://相对仓库根目录的路径)
</cite>

## 目录
1. [简介](#简介)
2. [项目结构](#项目结构)
3. [核心组件](#核心组件)
4. [架构总览](#架构总览)
5. [组件详细分析](#组件详细分析)
6. [依赖关系分析](#依赖关系分析)
7. [结论](#结论)

## 简介
...

## 项目结构
...

## 核心组件
...

## 架构总览
（如有架构图，用 Mermaid，图后附"图表来源"）

## 组件详细分析
...

## 依赖关系分析
...

## 结论
...
```

> 章节顺序可按实际情况增删，但"简介 → 项目结构 → 核心组件 → 架构总览 → 组件详细 → 依赖关系 → 结论"的递进逻辑建议保持。
>
> **分析深度要求（强制）**：`组件详细分析` 不应只是罗列表格（模块名/路径/职责），而应对主要组件给出**设计要点 / 关键流程 / 错误处理 / 风险控制**等维度的分析。判断是否"够深"的信号：若某节内容删除源码引用后仍能独立成立、仅是"该模块负责 XX"的常识性描述，则视为**深度不足**，必须补充"为什么这么设计/异常如何处理/有哪些风险"等设计层面内容。面向"总览/分类"的页面可把深度分析下沉到专门的子页，但总览页仍需用**有意义的行号区间**引用（覆盖被论述的函数/结构体，而非单点占位），且不得退化为纯目录式清单。
>
> **配图要求（强烈推荐，缺失需在交付摘要说明理由）**：涉及以下任一情形的页面**应配 Mermaid 图**，否则视为内容深度不达标（R7）：
> - **架构总览**：必须有架构/交互时序图
> - **控制流/状态机**：请求处理、任务调度、生命周期等流程必须有流程图
> - **数据结构/领域模型**：核心数据结构必须有类图
> - **多组件依赖**：组件间调用关系必须有依赖图
>
> 图表类型参考：架构图用 `graph`、时序交互用 `sequenceDiagram`、类与枚举用 `classDiagram`、判断分支用 `flowchart`。每个 Mermaid 图后必须紧跟 `图表来源`（见 R4）。
>
> **推荐补充章节（强烈推荐）**：以下章节能显著提升可用性，缺失时需在交付摘要中说明缺失原因：
> - `性能考虑`：针对该模块的并发/缓存/日志频率/路由开销等性能维度给出指导
> - `故障排查指南`：针对该模块常见故障给出排查路径（校验失败/参数无效/路由异常等）
> - `附录（设计模式/最佳实践）`：部署最佳实践、权限控制、扩展建议

**文档名称使用中文并带序号**：Wiki 文件名（即 `<page_path>` 的末级文件名，不含 `.md`）须使用中文且**带两位序号前缀**，序号按大纲/目录树中的页面顺序排列，便于检索与排序。格式为 `<序号>-<中文名>.md`，例如 `01-核心组件.md`、`02-依赖关系分析.md`；`##` 章节标题使用中文但**无需**序号。对应的目录锚点统一用中文（如 `[核心组件](#核心组件)`）。中文文件名更易检索、与索引构建的中文标题体系一致；序号保证同目录下页面稳定有序、避免歧义。

> 序号规则：从 `01` 起按大纲顺序排列；若后续增删页面导致序号不连续，应在交付摘要中说明，不强制重排已有页面序号（避免破坏已生成的引用/索引）。

### 引用与来源规范

- **`<cite>` 块**：文件顶部、标题之后，列出本页引用的所有源文件（路径摘要）。条目格式 `[名称](file://相对仓库根目录的路径)`，**仅需 `file://` 前缀，行号区间可省略**（cite 是路径清单，非精确引用）。
- **章节来源 / 图表来源**：每个章节后附 `章节来源`，每个 Mermaid 图后附 `图表来源`（亦接受 `图示来源`），列出支撑该节的源文件引用。条目格式 `[名称](file://相对仓库根目录的路径#Lx-Ly)`，**必须带 `file://` 前缀与行号区间**。
- **目录**：以 `## 目录` 标题起始，使用中文标题锚点，如 `[简介](#简介)`，且目录条目必须与 `##` 章节标题一一对应（`目录` 本身不计入条目）。

### 格式自检规则（R1-R6）

下表是格式规范的权威定义；实际交付门禁以 `codetowiki wiki-format` 退出码为准（见"格式检查脚本"）。

| 规则 | 要求 | 级别 | 不合规示例 |
|------|------|------|-----------|
| R1 | 顶部必须有 `<cite>` 块 | error | 缺 cite |
| R2 | `## 目录` 条目与 `##` 章节一致（`目录` 本身不计入） | error | 目录有但正文无该章节 |
| R3 | 每个章节后需有 `章节来源` | warning | 章节无来源 |
| R4 | Mermaid 图后需有 `图表来源`（亦接受 `图示来源`） | error | 有图无来源 |
| R5 | 所有引用路径必须 `file://` 前缀 | error | 用相对路径无前缀 |
| R6 | 来源块中源码引用需带 `#Lx-Ly` 行号区间，且区间应覆盖被论述代码的**实际跨度**（函数/结构体/方法体）；`Lx-Lx` 单点占位视为不合规（仅当确为一行关键代码、且无更宽上下文可引用时允许） | warning | 缺行号 / 单点占位 |
| R7 | **内容深度**：`组件详细分析`不得退化为纯描述性罗列（"XX 模块负责 YY"）；每个主组件须含≥2 个设计维度（设计要点/关键流程/错误处理/风险控制之一）；涉及架构/控制流/数据结构/多组件依赖的页面须配≥1 个 Mermaid 图 | warning | 全文无图 / 仅有"模块职责"式清单 |

**豁免说明**：
- **R3**：`目录` 章节豁免；纯概念性章节可写明"本节为概念性内容，不直接分析具体文件"（或"无章节来源"）以豁免，避免注入空来源块。
- **R6**：`<cite>` 块豁免（路径摘要无需行号）；目录、数据文件（如 `.dat`/`.json` 等非源码）豁免。
- **R7**：虽为 warning 级（不阻断 `codetowiki wiki-format` 退出码），但属**人工抽查不可豁免项**——缺失须实际补强（补图/补分析），不得以"无源码可引用"为由跳过。纯总览/分类页若确无控制流可经主 agent 判定后豁免配图，但须在摘要说明。

> **引用放置要求**：`章节来源` 必须紧跟其支撑的 `##` / `###` 小节，不得把所有引用堆在全文末尾的单一块里；每个子节的来源只列该节真正用到的文件，避免"全量罗列"式凑数。Mermaid 图后必须紧跟 `图表来源`（见 R4）。

> 机械修复只改格式，不得凭空改正文含义。error 级阻断交付，warning 级尽力补全。

### 高质量 Wiki 特征基准（内容质量对标）

> R1-R7 是**门禁下限**（必须通过）；本表是**质量对标目标**（应主动靠近）。两者互补：门禁拦住"不合格"，基准引导"高质量"。子 agent 撰写时对标本表，主 agent 汇总时据此抽查。

格式校验（R1-R7）是下限门禁；以下是"高质量 Wiki"的内容特征，子 agent 撰写时应主动对标，主 agent 汇总时据此抽查：

| 特征 | 最低要求 | 反例（不合格信号） |
|------|----------|--------------------|
| 章节完整度 | 含简介/项目结构/核心组件/架构总览/组件详细/依赖/结论 + ≥2 个推荐章节（性能/故障/附录） | 仅 4-5 节、无推荐章节 |
| Mermaid 图覆盖 | ≥2 个图，覆盖架构 + 至少一个流程/数据结构 | 全文 0 图 |
| `###` 子节深度 | `组件详细分析` 下按组件拆 `###`，每节含设计要点+关键流程 | 全文无 `###`，只有 `##` 平铺 |
| 分析维度 | 设计要点/关键流程/错误处理/风险控制 四维度至少命中 2 个 | 仅"该模块负责 XX"式描述 |
| 故障排查 | 针对模块实际校验失败/路由异常等给出排查路径 | 通用空话如"检查日志" |
| 引用密度 | 每个主组件有独立行号区间引用，cite 覆盖该模块全部源文件 | 全文仅 3-5 个文件引用 |

> 汇总阶段若发现某页明显偏离基准（如 0 图、无 `###`、仅描述性内容），即使 R1-R7 通过也应退回子 agent 补强，并在交付摘要标注需返工的页面。

### 格式检查脚本（强制）

CodeToWiki 提供 `codetowiki wiki-format` 命令，覆盖 R1-R6 全部规则（R7 内容深度为人工抽查项，见"高质量 Wiki 特征基准"）。**每生成一篇 Wiki 页面后，必须立即调用该命令校验**，未通过（退出码非 0）不允许交付，需修复后复检直至通过。

```bash
# 检查单篇（生成后立即执行）
codetowiki wiki-format --file <生成的wiki.md>
# 退出码：0=合规；1=存在 error（或 --strict 时存在 warning）；2=运行异常（如文件无法读取/解析失败）

# CI 门禁：递归检查整个 wiki 目录
codetowiki wiki-format --wiki-dir <wiki_root>

# 警告也视为失败
... --strict

# 机器可读输出（便于流水线解析）
... --json

# 自动修复可机械修复项（R1/R2/R3/R4/R5；R6 行号与 R7 内容深度需人工补全），修复后须复检
codetowiki wiki-format --fix --file <生成的wiki.md>
```

**调用纪律**：
- 单页：主 agent 写完即跑 `--file` 校验。
- 多页 / 分批：每个子 agent 写完其页面后，主 agent 或子 agent 需对该页跑 `--file` 校验；全部批次结束再跑一次 `--wiki-dir` 全量校验。
- `--fix` 仅做机械修复（补 `<cite>`/来源块、加 `file://`），**不得替代人工核对正文与引用准确性**，也**无法修复 R7 内容深度**（需人工补图/补分析）；修复后必须复检确认退出码为 0。
- 校验发现 error 级问题（R1/R2/R4/R5）必须修复；warning 级（R3/R6/R7）应尽力补全来源、行号与内容深度，仅在确无源码可引用时可在摘要中说明并豁免 R3/R6（R7 内容深度不可豁免，须实际补强）。

## 标准流程

### Step 1: 确认范围、存储位置与排除项

按"前置确认"与用户对齐，记录：Wiki 根目录（绝对路径）、分析范围、排除路径、拆分粒度、语言。

### Step 2: 代码调研（受排除项约束）

- 在**排除路径之外**，用 grep / 目录浏览 / 语义搜索理解目标代码的功能定位、核心类/函数、调用关系。
- 对候选"新功能/独立模块"，读取 2-3 个核心文件快速理解（跳过 `__init__.py`、`urls.py` 等胶水文件）。
- 记录关键文件路径与行号，供后续引用。

### Step 3: 生成大纲（Outline）并落盘

产出结构化大纲，包含：

- **目录树**：顶层目录 + 各子页路径
- **每页条目**：标题、目标路径（相对 Wiki 根）、引用的源文件清单（含行号）、章节骨架（简介/项目结构/核心组件/…）

写入大纲文件（持久化，避免长对话上下文丢失）：

```
<wiki_root>/.wiki-plan/outline.md
```

> 该文件**不计入 Wiki**，生成完成或确认后建议删除，且不得被后续索引工具扫描。它作为"规划契约"，供 Step 6 并行撰写时各子 agent 读取。

### Step 4: 用户确认大纲（必须）

- 向用户展示大纲：目录树 + 各页摘要 + 引用源文件清单 + 排除项。
- 用户可调整目录结构、增删页面、改范围或排除项。
- ⚠️ **未确认前禁止开始写任何 Wiki 正文**。大纲是正式撰写的唯一依据。

### Step 5: 体量评估与分批排期

在撰写前，先统计本次任务的体量（数据在 Step 2/3 已可得）：

- `页面数`：大纲中的 Wiki 页面总数
- `源文件数`：大纲各页引用的源文件去重计数（已排除排除项）

**分批阈值**（默认，可按项目调整；满足任一即启用分批）：

| 指标 | 阈值 |
|------|------|
| 页面数 | > 8 |
| 源文件数 | > 30 |

- 未超阈值：直接进入 Step 6 一次性撰写。
- 超阈值 → **必须先排期，再分批撰写**（强制流程，不可跳过）：
  1. 将页面按领域 / 目录 / 依赖关系聚为若干批（建议每批 3-5 页，同批页面共享上下文、减少跨批引用）。
  2. 确定批次顺序（如：先基础/核心模块，后上层业务；或依赖从底向上）。
  3. 输出**排期表**给用户：批次号、每批页面、预计涉及源文件数。
  4. 用户确认排期后，逐批执行 Step 6；每批完成后调用格式检查命令逐页校验，全部批次结束再统一汇总。

> ⚠️ 体量超阈值时，未排期不得开始撰写。一次性生成大量页面会导致上下文溢出、质量下降，分批是强制要求而非可选项。

### Step 6: 细化撰写（按大纲/分批，并行加速）

- **单页（页面数 == 1）**：主 agent 直接按"页面结构模板"撰写，写完后**立即调用格式检查命令**（`--file`）校验，未通过则修复复检。
- **多页（页面数 ≥ 2）**：调用 **task-dispatch** skill，将每个页面作为一个独立子任务并行分派给子 agent：
  - 每个子 agent 输入：大纲中该页的条目 + 本 SKILL 的"Wiki 格式规范" + 需引用的源文件路径 + 格式检查命令调用要求。
  - 子 agent 职责：读取源文件 → 按模板撰写该页 → 写入 `<wiki_root>/<page_path>` → **调用格式检查命令**（`--file` 校验该页）确认退出码为 0，未通过则修复复检。
  - 主 agent 职责：汇总各页、统一风格、补 `[待审核]` 标记，并对全部页面跑一次 `--wiki-dir` 全量校验。
- **分批模式**：若 Step 5 启用了分批，按排期表逐批执行本步骤；每批内部仍可用 task-dispatch 并行，**每批结束后对该批各页逐个跑 `--file` 校验**，全部批次结束再跑一次 `--wiki-dir` 全量校验。
- 所有新页面首行加：`<!-- [待审核] AI 自动生成，请人工确认后移除此标记 -->`
- **禁止脚本自动生成正文**：子 agent 也须读取源码后手动分析撰写，不得用脚本批量生成。

### Step 7: 格式自检与落盘（必须调用脚本）

- **强制调用格式检查命令**：对全部生成的页面执行 `codetowiki wiki-format --wiki-dir <wiki_root>`（或逐页 `--file`），确认退出码为 0。
  - 存在 error 级问题（R1/R2/R4/R5）必须修复；warning 级（R3/R6/R7）应尽力补全，确无源码可引用时在摘要说明。
  - 可加 `--strict` 让 warning 也视为失败，作为更严格的门禁。
  - 机械修复用 `--fix`（仅补格式块与 `file://` 前缀），修复后必须复检至退出码 0。
- **内容深度抽查（R7）**：格式脚本通过后，按"高质量 Wiki 特征基准"抽查每页——是否配图、是否有 `###` 子节深度分析、是否含推荐章节。明显偏离基准的页面（如 0 图、仅描述性罗列）退回子 agent 补强，不得直接交付。
- 逐条核对 R1-R7 的同时，以命令输出为准；命令未通过不得交付。
- 确认写入用户指定的 Wiki 根目录。
- 清理或保留 `.wiki-plan/outline.md`（建议完成后删除，避免污染索引）。

### Step 8: 完成摘要

输出：
- Wiki 根目录
- 新建页面数（含路径）
- 拆分结构说明
- 排除项清单
- 是否启用分批（含排期表与批次数）
- 是否标注 `[待审核]`
- 格式校验结果（`codetowiki wiki-format` 退出码 0 / R1-R6 通过情况；无命令时为手动自检结果）
- 大纲文件去向（已删除 / 保留路径）

## AI 生成原则

| 原则 | 要求 |
|------|------|
| 大纲先行 | 先出大纲并落盘，再细化；不跳过规划直接写正文 |
| 先确认后写 | 大纲须经用户确认，未确认不得写正文 |
| 排除项约束 | 调研与建页都跳过排除路径，避免全盘扫描与噪音页 |
| 分批机制 | 体量超阈值（页面>8 或 源文件>30）必须分批；分批前先排期并经用户确认 |
| 引用可追溯 | 每个论断必须能对应到源文件引用（file:// + 行号），且行号区间须覆盖实际分析跨度，禁止单点占位 |
| 内容深度对标 | 不得退化为纯描述性罗列；组件须含设计要点/流程/风险等维度；架构/控制流/数据结构页须配 Mermaid 图（R7 + 高质量基准） |
| 最小杜撰 | 只描述代码实际行为，不臆测未实现的意图 |
| 保留人工内容 | 若目标目录已有 Wiki，保留人工补充，仅增量扩展 |
| 并行加速 | 多页面时通过 task-dispatch 分派子 agent，主 agent 汇总校验 |
| 新建标注审核 | AI 生成页必须标注 `[待审核]` |
| 格式校验必做 | 每生成一篇 Wiki 必须做格式校验：安装 CodeToWiki 时强制调用 `codetowiki wiki-format`（退出码非 0 不得交付）；无命令时按 R1-R6 手动自检 |
| 存储位置必确认 | 未确认 Wiki 根目录前不得写入任何文件 |
